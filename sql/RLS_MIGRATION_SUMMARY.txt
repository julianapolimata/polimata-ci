3. SECURITY POLICIES (Per Table)
   For each table: SELECT, INSERT, UPDATE, DELETE policies
   Policy strategy varies by role and data sensitivity:

   PERFIS (Users):
   - SELECT: Own profile OR admin OR same client
   - UPDATE: Own profile OR admin
   - INSERT/DELETE: Admin only

   CLIENTES (Tenants):
   - SELECT: Admin OR own client
   - INSERT/UPDATE/DELETE: Admin only

   PROJETOS (Projects):
   - SELECT: Admin OR own client OR assigned project
   - INSERT: Admin, gestor_cliente
   - UPDATE: Admin OR gestor_cliente (own client)
   - DELETE: Admin only

   AREAS (Areas within projects):
   - SELECT: Admin OR projects owned/assigned
   - INSERT/UPDATE: Admin, gestor_cliente, consultor_polimata
   - DELETE: Admin only

   MRC (Risk/Control Matrix - MOST CRITICAL):
   - SELECT: Admin OR projects owned/assigned
   - INSERT/UPDATE: Admin, gestor_cliente, consultor_polimata
   - DELETE: Admin only

   REVISOES (Reviews):
   - SELECT: Admin OR projects owned/assigned
   - INSERT/UPDATE: Admin, gestor_cliente, consultor_polimata
   - DELETE: Admin, gestor_cliente

   NOTIFICACOES (Notifications):
   - SELECT: Admin OR global OR projects owned/assigned
   - INSERT: Admin, gestor_cliente, consultor_polimata
   - UPDATE/DELETE: Admin only

   SISTEMAS (Systems):
   - SELECT: Admin OR own client
   - INSERT/UPDATE: Admin, gestor_cliente
   - DELETE: Admin only

   RESPONSAVEIS (Responsible people):
   - SELECT: Admin OR own client
   - INSERT/UPDATE: Admin, gestor_cliente
   - DELETE: Admin only

   CONSTANTES (Global constants):
   - SELECT: All authenticated users
   - INSERT/UPDATE/DELETE: Admin only

4. JUNCTION TABLES (if exist)

================================================================================
ROLE ACCESS MATRIX
================================================================================

                    SELECT  INSERT  UPDATE  DELETE  COMMENT
admin_polimata      ALL     ALL     ALL     ALL     Full system access
gestor_cliente      CLIENT  CLIENT  CLIENT  -       Client's data only
consultor_polimata  PROJ    PROJ    PROJ    -       Assigned projects only
usuario_cliente     CLIENT  -       -       -       Read-only client access

Note: PROJ = projects they're assigned to (via perfis_projetos)
      CLIENT = all data for their assigned client_id
      ALL = all system data

================================================================================
ISOLATION KEYS
================================================================================

Primary:  cliente_id (on perfis, clientes, sistemas, responsaveis)
Secondary: projeto_id (on projetos, areas, mrc, revisoes)
Tertiary: perfil_id → projeto_id (perfis_projetos junction)

Data flow: auth.uid() → perfis.cliente_id → clientes/projetos
                              ↓
                          projetos.cliente_id → areas/mrc/revisoes

================================================================================
MULTI-TENANCY GUARANTEE
================================================================================

✓ User data is scoped by cliente_id
✓ Project data is scoped by projeto_id (which belongs to cliente_id)
✓ Cross-client data access is impossible (even with SQL injection)
✓ Cross-project data access restricted by role & perfis_projetos
✓ All isolation enforced at DB level (RLS) not application level
✓ Even Supabase operators cannot bypass (RLS is DB-native)
