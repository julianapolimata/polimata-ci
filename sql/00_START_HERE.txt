Part B: Test each role
  Follow instructions in DEPLOYMENT_GUIDE.md section "Testing the Migration"

  Test admin role:
    - Log in as admin_polimata user
    - Run: SELECT COUNT(*) FROM mrc;
    - Should return a number (data visible)

  Test client manager role:
    - Log in as gestor_cliente user
    - Run: SELECT COUNT(*) FROM mrc;
    - Should return only data for their client

  Test consultant role:
    - Log in as consultor_polimata user
    - Run: SELECT COUNT(*) FROM mrc;
    - Should return only data from assigned projects

  Test client user role:
    - Log in as usuario_cliente user
    - Run: SELECT COUNT(*) FROM mrc;
    - Should return data (read-only)
    - Run: INSERT INTO mrc (...) VALUES (...);
    - Should FAIL with permission error

================================================================================
COMMON QUESTIONS ANSWERED
================================================================================

Q: Will this break my existing application?
A: No. RLS works transparently. Your code doesn't need changes. The database
   will just reject requests that violate the security policies.

Q: What if I made a mistake?
A: Don't panic. You can disable RLS on all tables (see DEPLOYMENT_GUIDE.md
   "Rollback Plan"). The migration uses transactions, so either all policies
   are created or none are.

Q: How long does it take to run?
A: ~5-10 seconds total. No downtime needed.

Q: What if a user can't see their data after deployment?
A: Most likely cause: Their user record in the perfis table is missing the
   correct cliente_id or papel. Check DEPLOYMENT_GUIDE.md "Common Issues".

Q: Can I test this without affecting production?
A: YES PLEASE. Deploy to staging first. See Step 2 above.

Q: What's the performance impact?
A: Minimal with indexes. See Step 6 above to add recommended indexes.

Q: Can I roll this back?
A: Yes. See DEPLOYMENT_GUIDE.md "Rollback Plan" section.

================================================================================
WHAT THIS PROTECTS
================================================================================

After deployment, your database is protected against:

✓ Cross-client data access (user sees only their client's data)
✓ Cross-project data access (consultant sees only assigned projects)
✓ Unauthorized modifications (usuario_cliente cannot edit)
✓ SQL injection (RLS enforced at database level)
✓ Privilege escalation (users cannot change their own role)
✓ Admin impersonation (all operations checked against perfis table)

Even if someone gains direct database access, they can only see data that
their user role is allowed to see.

================================================================================
IF SOMETHING GOES WRONG
================================================================================

Error 1: "relation does not exist"
  → A table mentioned in the SQL doesn't exist
  → Create the table first, then re-run the SQL
  → Or contact your DBA

Error 2: "Permission denied"
  → Your database user doesn't have DDL permissions
  → Use a superuser or role with CREATE/ALTER permissions
  → Contact your Supabase admin

Error 3: "column does not exist"
  → A table structure is different than expected
  → Check DEPLOYMENT_GUIDE.md "Common Issues"
  → Compare your schema with expected tables

Error 4: Queries return no rows after deployment
  → RLS policy is filtering out data
  → Check user's cliente_id and projeto_id match the data
  → See DEPLOYMENT_GUIDE.md "Testing" for debugging

Error 5: Junction tables error
  → Tables perfis_projetos or permissoes_area don't exist
  → The migration handles this gracefully (IF EXISTS checks)
  → Create those tables if you need them, then re-run SQL
