Trilha de Desenvolvimento — Cada fase tem seu ciclo de aprovação
Diagnóstico Inicial
Avaliação binária: o controle existe ou não
Peso: 10% (fixo por área)
Ciclo completo desta fase (exemplo detalhado)
Rascunho→ consultor preenche→ resultado + ficha→ Em Revisão→ Aprovado→ Avança p/ próxima fase ▼
↩ Quando o admin reprova
1
Admin reprova com nota obrigatória
Status muda de Em Revisão para Reprovado
Consultor recebe notificação (sistema + email) com o motivo
2
Consultor abre o controle e vê a nota de reprovação
Pode editar todos os campos da análise (resultado, inconsistência, recomendação, etc.)
O controle permanece na mesma fase — não regride nem avança
3
Consultor clica "Salvar e Reenviar para Revisão"
Status muda de Reprovado para Em Revisão
Admin recebe nova notificação de submissão
4
Admin revisa novamente → Aprovar ou Reprovar
Se aprovar: controle avança para a próxima fase (status_workflow reseta para Rascunho na nova fase)
Se reprovar novamente: volta ao passo 1. O ciclo se repete sem limite de tentativas
Em Revisão → ↩ Reprovado → consultor edita → resubmete → Em Revisão → ↩ Reprovado → ... (repete) → Aprovado → Avança ▼
O loop reprovação → edição → resubmissão pode ocorrer quantas vezes for necessário. Cada iteração fica registrada no histórico com fase, data e nota.
Regras durante reprovação
● Controle reprovado NÃO entra no cálculo de maturidade (contribui 0%)
● Cliente vê o controle como Em Análise — nunca vê "Reprovado"
● Somente o consultor que submeteu (e o admin) veem a nota de reprovação
● Consultor pode "Salvar Alterações" sem reenviar (mantém status Reprovado para continuar editando depois)
Todas as fases abaixo seguem o mesmo ciclo de aprovação (com loop de reprovação) mostrado acima

Regra de regressão (calculoMaturidade.js)
● Resultado Inefetivo ou GAP em qualquer fase a partir de F2-E2 → regressão total para F2-E1
● Todas as contribuições acumuladas são zeradas (contribuição do controle = 0%)
● Se veio do atalho (F1 Efetivo), o atalho é revogado — agora segue caminho normal
● F1 é exceção: Inefetivo/GAP na F1 não causa regressão — apenas define o caminho normal (sem atalho)

Como o workflow de aprovação se conecta com a Trilha
O resultado do teste (Efetivo / Inefetivo / GAP) determina se o controle avança ou regride na Trilha. Mas essa movimentação só acontece quando o admin aprova a análise. Ou seja: o consultor registra "Inefetivo" → submete → admin revisa e aprova → aí sim o sistema aplica a regressão e recalcula a maturidade. Enquanto não for aprovado, o resultado não tem efeito no cálculo.
